# Bun + Hono 代理 Client Dist 最小 Demo 设计

## 1. 背景与目标

当前仓库已经具备基础 monorepo 结构：

- `apps/client`：Vite + React 前端应用
- `apps/server`：Bun + Hono 服务端应用

本次目标是在不引入额外复杂度的前提下，补齐一个最小可运行 demo，验证以下链路：

1. 前端可以正常本地开发
2. 服务端可以提供最小 API 给前端调用
3. 前端构建后的 `dist` 可以由服务端直接托管
4. 同一套前端代码在开发态和生产态都使用 `/api` 路径访问后端

该 demo 的定位是模板起点，不是完整业务骨架。

## 2. 范围

### 2.1 本次包含

- 服务端提供三个最小 API：
  - `GET /api/health`
  - `GET /api/hello`
  - `POST /api/echo`
- 前端首页在加载时请求 `GET /api/hello`
- 前端提供一个最小表单，请求 `POST /api/echo`
- 开发态通过 Vite 代理 `/api` 到服务端
- 生产态由服务端直接托管 `apps/client/dist`
- 提供最小使用文档，说明开发态与生产态如何运行

### 2.2 本次不包含

- 数据库
- 登录鉴权
- SSR
- 前后端共享 SDK
- 复杂状态管理
- 前端路由系统扩展
- 部署脚本与容器化
- 复杂错误页与监控埋点

## 3. 方案选择

### 方案 A：生产态托管 + 开发态代理

- 生产态中，服务端托管前端 `dist`
- 开发态中，前端继续使用 Vite dev server，Vite 代理 `/api` 到服务端

优点：

- 结构清晰，职责稳定
- 前端代码里不需要区分开发和生产 API 地址
- 实现复杂度最低，同时覆盖两种运行方式

缺点：

- 开发态入口仍然分为前后端两个进程

### 方案 B：开发态和生产态全部由服务端统一入口

优点：

- 入口统一

缺点：

- 当前模板需要补更多开发期转发逻辑
- 对“先做最简单 demo”不友好

### 方案 C：只做生产态

优点：

- 代码最少

缺点：

- 不能满足“开发态和生产态都要”的要求

### 结论

采用方案 A：生产态由 `server` 托管 `client/dist`，开发态由 Vite 代理 `/api` 到 `server`。

## 4. 架构设计

### 4.1 模块职责

#### `apps/client`

- 负责页面渲染
- 使用浏览器 `fetch` 请求 `/api/*`
- 不感知后端真实地址
- 构建输出到 `dist`

#### `apps/server`

- 提供最小 API
- 在生产态托管 `apps/client/dist`
- 对非 `/api` 请求执行静态资源返回或 `index.html` 回退

### 4.2 目录与职责边界

建议保持现有目录不变，仅补最小代码：

- `apps/client/src/App.tsx`
  - 展示 `hello` 接口结果
  - 提交 `echo` 请求
- `apps/client/vite.config.ts`
  - 配置 `/api` 代理到本地服务端
- `apps/server/src/index.ts`
  - 定义 API 路由
  - 定义静态托管和 SPA fallback
- `apps/server` 下新增测试文件
  - 覆盖 API 行为
- 根目录与各子包 `package.json`
  - 增加最小运行脚本

## 5. 数据流设计

### 5.1 开发态

1. 浏览器访问 Vite dev server
2. 前端页面发起 `/api/hello` 和 `/api/echo`
3. Vite 将 `/api/*` 代理到 `apps/server`
4. 服务端返回 JSON
5. 前端展示结果

这样前端无需维护环境分支逻辑，开发与生产请求路径保持一致。

### 5.2 生产态

1. 先执行 `client build`
2. 服务端读取并托管 `apps/client/dist`
3. 浏览器访问服务端地址
4. `/api/*` 由 Hono 处理
5. 静态资源路径直接返回文件
6. 非 API 且未命中静态资源的路径回退到 `index.html`

这样后续若前端增加路由，也不会因为刷新页面而 404。

## 6. API 设计

### 6.1 `GET /api/health`

用途：

- 健康检查
- 确认服务端已启动

示例响应：

```json
{
  "ok": true,
  "service": "server",
  "time": "2026-04-16T00:00:00.000Z"
}
```

### 6.2 `GET /api/hello`

用途：

- 给首页首屏展示一条来自服务端的文案
- 验证前后端已联通

示例响应：

```json
{
  "message": "Hello from server",
  "time": "2026-04-16T00:00:00.000Z"
}
```

### 6.3 `POST /api/echo`

用途：

- 演示最基础的 JSON 提交与回显

请求体：

```json
{
  "message": "hello demo"
}
```

成功响应：

```json
{
  "message": "hello demo",
  "echoedAt": "2026-04-16T00:00:00.000Z"
}
```

失败规则：

- 请求体不是合法 JSON，返回 `400`
- `message` 缺失、为空字符串或不是字符串，返回 `400`

## 7. 前端交互设计

首页只做最小演示，不引入复杂 UI。

页面状态包括：

- `hello` 数据加载中
- `hello` 加载成功
- `hello` 加载失败
- `echo` 提交中
- `echo` 提交成功
- `echo` 提交失败

页面元素包括：

- 标题与说明文字
- 服务端 `hello` 响应展示区
- 一个输入框，用于输入 `message`
- 一个提交按钮，用于请求 `POST /api/echo`
- 一个结果展示区，用于显示回显结果或错误信息

前端请求路径统一写为相对路径：

- `/api/hello`
- `/api/echo`
- `/api/health`

## 8. 错误处理

### 8.1 服务端

- 未匹配 API 返回 `404`
- 无效 JSON 返回 `400`
- 参数校验失败返回 `400`
- 静态文件缺失时：
  - 如果是前端路由访问，则回退到 `index.html`
  - 如果是明确的资源路径但文件不存在，则返回 `404`

### 8.2 前端

- 请求失败时展示简单错误文案
- 不做自动重试
- 不接入全局错误边界

## 9. 测试策略

遵循本仓库要求，优先做最小 TDD 闭环。

### 9.1 服务端测试

先写失败测试，再补实现。至少覆盖：

- `GET /api/health` 返回 `200` 和预期字段
- `GET /api/hello` 返回 `200` 和预期字段
- `POST /api/echo` 在合法请求下返回 `200` 和回显内容
- `POST /api/echo` 在非法请求下返回 `400`

测试重点放在行为，不测试内部实现细节。

### 9.2 前端验证

本次不引入前端测试框架。

采用手动验证：

- 开发态下页面可正常请求 `/api`
- 生产态下服务端可托管 `dist`
- 页面可展示 `hello`
- 页面可提交 `echo`

## 10. 脚本与运行方式

### 10.1 开发态

目标：

- 能并行启动 `client` 和 `server`
- 前端通过代理访问后端

建议：

- `client` 保留现有 `dev`
- `server` 保留或补全 `dev`
- 根目录增加一个并行开发脚本，作为模板统一入口

### 10.2 生产态演示

目标：

1. 先构建前端
2. 再启动服务端
3. 通过单一端口访问页面和 API

建议最小流程：

1. 执行前端构建
2. 启动服务端生产模式
3. 访问服务端地址验证页面和接口

## 11. 实施顺序

1. 为服务端 API 编写失败测试
2. 实现服务端 API
3. 实现服务端静态托管与 SPA fallback
4. 配置前端开发态代理
5. 改造前端页面，接入 `hello` 与 `echo`
6. 增加根目录和子包运行脚本
7. 补最小 README 使用说明
8. 运行测试与手动验证

## 12. 验收标准

满足以下条件即可视为完成本次 demo：

1. 开发态下能同时启动前后端
2. 前端开发页请求 `/api/hello` 成功
3. 前端表单提交 `/api/echo` 成功
4. 前端构建后，服务端能直接托管 `client/dist`
5. 生产态访问单一服务端端口即可打开页面并访问 API
6. 服务端最小测试通过

## 13. 风险与约束

- `server` 目前基于 Bun，测试与生产启动方式要优先兼容 Bun 运行时
- 静态托管路径需要从 `apps/server` 正确定位到 `apps/client/dist`
- 当前仓库尚未补齐完整脚本，实施时需避免引入过重依赖

- 增加前端路由
- 增加更完整的 API 错误模型
- 引入前端测试
